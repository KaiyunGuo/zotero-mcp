本文档由 superpowers:brainstorming 流程产出。 设计第 1–6 段全部 approved。 设计阶段完成。 下一步:写 spec → 自检 → Yun 评审 → 转 writing-plans skill。 完整决策与下一步见同目录 HANDOFF.md。

1. 目标与动机

现有的第三方包 zotero-mcp-server (0.3.0) 功能受限:约 5 个工具绕过配置而失效、 子笔记挂不上(需 Web API 凭证)、不能触发 Zotero 的「Find Available PDF」、DB 路径自动探测在本机失效。 其根因是 它同时用 SQLite 直读和 Local API 两条通道,两者配置不一致。

Yun 的决定:不修补现有包,而是从零自建一个完全掌控的 zotero-mcp, 按自己的文献工作流长期演进。核心工作流为:按论文名检索 → 入库 Zotero → 抽取全文 → 把阅读笔记写回 Zotero 条目(在 UI 中可见)。

2. 已锁定的决策

D4连接通道 = Local API only单一代码路径;消灭"DB 路径探测 + SQLite schema 解析"这一整类拖垮旧包的 bug
D5v1 范围 = 全部 4 个能力包 + Find Available PDF检索/元数据/全文 + 入库 + 笔记读写 + 组织
D6架构 = 分层:server.py → client.py → pyzotero工具体极薄;领域逻辑集中在 ZoteroClient
D7v1 不引入 models.py 数据类No premature abstractions

D18日志默认 = WARNING(stderr)env ZOTERO_MCP_LOG=INFO/DEBUG 切换。stdout 严禁写日志(留给 MCP JSON-RPC)
D19测试 = 两层金字塔(unit + live integration);不建 mock HTTP fixture 层单用户、Zotero 永远在跑;mock 漂移代价 > 测试快收益。允许个别 error-path 用例 inline 用 responses 打补丁(网络超时、5xx、畸形 JSON 等真 Zotero 难触发的病理路径),不抽公共 fixture
D20测试隔离 = dedicated collection __zotero_mcp_test__ + __test__ tag + teardown共用日常 library;不上专用 Zotero profile(YAGNI;profile 漂移反而掩盖真实 bug)
D21TDD 姿态 = 混合(纯逻辑严格 TDD / client.py 测试伴随 / server.py 滞后到 e2e)按文件性质分:纯函数零摩擦严格 TDD;client.py 严格 TDD 会被 Zotero 状态拖死;server.py 薄壳 unit 测 mock 价值低,e2e 更划算
D22v1 不上 CI;本地 git pre-commit hook 跑 pytest tests/unit/单用户私库,CI 的协作把关价值不存在;pre-commit 比 CI 早一步拦截倒退。未来转公开/多人时再加 unit-only CI

3. 方案对比(为何选 Local API only)

方案说明结论
B — Local API only所有读写都走 localhost:23119/api/✓ 已选 单路径;消灭旧包栽过的 bug 类;子笔记原生支持
A — HybridSQLite 直读 + Local API 写✗ 否决 两套代码路径;重引入 SQLite 路径探测+schema 解析

7. 错误处理(设计第 4 段 — ✅ approved 2026-05-15)

7.1 错误分类(errors.py)

ZoteroMCPError                      # 基类
├─ ZoteroNotRunningError            # 连不上 localhost:23119
├─ ZoteroVersionError               # Zotero < 7.0(写操作)
├─ LocalAPIError                    # 通用 HTTP
│   ├─ ItemNotFoundError            # 404
│   ├─ ValidationError              # 400 / 412
│   └─ ServerError                  # 5xx
├─ TranslatorError                  # translator 解 DOI/URL 失败
├─ BBTUnavailableError              # 仅 search_by_citekey 显式需要
└─ FindPDFNotSupportedError         # spike 结论:API 不可触发

7.2 预检的两层架构

层时机内容缓存
启动预检server 启动一次① Local API 可达 ② Zotero 版本 ③ BBT JSON-RPC ④ (spike 后) find_pdfcapabilities 字典,写 stderr
每调用预检每次 tool 调用入口一次便宜的 HTTP ping;失败则重跑启动预检不缓存

启动日志样例:[zotero-mcp] zotero_running=True version=9.0.x bbt_available=True find_pdf=<pending spike>

7.3 HTTP 错误映射

HTTP / 异常映射成用户可见 hint
ConnectionRefused / TimeoutZoteroNotRunningError"Zotero is not running on localhost:23119. Please start Zotero."
404ItemNotFoundError"Item not found: <key>"
400 / 412ValidationError透传 Zotero 错误消息
403LocalAPIError"Local API may require enabling in Zotero → Settings → Advanced"
5xxServerError"Zotero internal error; retry or restart Zotero"

7.4 ⭐ Return vs Raise 纪律

类情况
Raise("动作未完成")Zotero 没开 / item 不存在 / 版本 < 7.0(D17)/ 校验失败 / 5xx / BBT 显式必需但未装
Return("动作完成,结果信息丰富")撞重复(D15);find_pdf 没找到;全文未索引;BBT 未装时 citekey=null
规则:raise = 动作未完成;return = 动作完成,结果信息丰富。 把"撞重复"错抛会让 LLM 误判失败盲目重试;把"Zotero 没开"当 return 会让 LLM 拿空数据继续。

7.5 Zotero 版本门(D17)

  • 启动预检读到 version < 7.0 → 所有写工具不可用
  • 调写工具时直接 raise ZoteroVersionError(硬抛,无宽容模式)
  • 读工具不受影响——6.x 也能读
  • 本机 v9 远超门槛,这条门主要作未来兜底

7.6 日志(D18)

  • stdout 留给 MCP JSON-RPC——绝不写日志到 stdout
  • 所有日志走 stderr,默认 WARNING
  • env ZOTERO_MCP_LOG=INFO/DEBUG 切换
  • 启动时无条件写一次 capability 行

8. 测试策略(设计第 5 段 — ✅ approved 2026-05-19)

8.1 两层金字塔(D19)

层位置性质速度
unittests/unit/纯函数,无 Zotero,无 HTTP全跑 < 1s
live integrationtests/integration/真打 localhost:23119,每个 MCP 工具至少 1 个 smoke全跑数秒

不建立通用 mock HTTP fixture / cassette 层。砍掉的理由:单用户、Zotero 永远在跑,mock 与真 Zotero 形状漂移的代价大于"测试快"的收益,且会培养"绿测 = 安全"的错觉。 但允许在个别 error-path 测试里 inline 使用 responses 库打补丁(网络超时、服务端 5xx、畸形 JSON 等真 Zotero 难触发的病理路径),不抽取公共 fixture。

8.2 测试隔离(D20)

共用 Yun 日常 Zotero library——不上专用 Zotero profile(YAGNI;profile 漂移反而掩盖真实 bug)。三层防护:

  1. 所有测试创建的 item / note 进固定 collection __zotero_mcp_test__(conftest.py session fixture 启动时确保存在)
  2. 每个测试对象打 __test__ tag,便于事后人工清理残留
  3. 用例 teardown 删除自己创建的 item;最终残留靠 tag 兜底批量清理

查询类工具的断言里加 collection / tag 过滤,避免日常库真数据干扰。

8.3 TDD 姿态(D21)

文件性质姿态
formatting.py / errors.py / config.py纯函数严格 TDD:先红 → 绿 → 重构
client.py领域逻辑 + I/O测试伴随:每个方法 land 前必须有 ≥1 integration smoke
server.pyMCP 工具薄壳滞后:client 方法都覆盖后再补 e2e workflow 测试

8.4 capability-gated 测试

BBT / find_pdf 相关测试在 conftest.py 探测 capability flag;不可用时 pytest.skip(...)。和 §7.2 启动预检共用同一份探测逻辑(实现层面提取一个 probe_capabilities(),启动和测试都调用)。

8.5 CI 策略(D22)

  • v1 不配 GitHub Actions——单用户、私库,CI 的协作把关价值不存在
  • 本地 git pre-commit hook 跑 pytest tests/unit/,commit 前拦截倒退
  • integration 全程手动跑(pytest tests/integration/)
  • 未来项目转公开 / 多人时再加 CI(仅 unit)

11. 风险细化(设计第 6 段 — ✅ approved 2026-05-19)

对每一项 spike,锁定:何时做 · 验什么 · 通过判据 · 不通过的 Plan B。

11.1 Spike 1 — find_pdf 可行性 ⭐(D12 关键路径)

何时写代码前 最优先(决定 add_by_doi/url 默认 find_pdf_fallback=True 是否站得住)
验什么Zotero v9 是否暴露 Local API 端点能从外部触发"Find Available PDF"(等价于 UI 右键菜单那条动作)

Plan B端点不存在 → zotero_find_pdf 工具保留但 raise FindPDFNotSupportedError(§7.1 已为此准备);add_by_doi/url 默认 find_pdf_fallback=False;§6 数据流图删去 ②.2 兜底支
